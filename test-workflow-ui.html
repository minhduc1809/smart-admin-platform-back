Workflow System Test UI

Simulates a Leave Request workflow with multiple roles: Employee submits, Manager/HR reviews, and the system tracks all state changes.

Phase 1 — Setup

1. Login (Get Token)

Who: Any user (Employee, Manager, or Admin)
What: Authenticate and get a JWT token. The token determines your role and what actions you can perform.
Tip: Use admin@example.com (ADMIN role) to test all actions, or create separate users with MANAGER/USER roles for realistic testing.

2. Create Workflow Definition

Who: Admin or Manager (one-time setup)
What: Define the approval process for a form. This config tells the system: who can approve, what states exist, and which transitions are allowed.
When: Once per form type. After creation, all submissions to this form will follow this workflow automatically.
Workflow Flow: Employee submits → [pending_manager] Manager approves → [pending_hr] → HR approves → [approved] Manager rejects → [rejected] Manager returns → [returned_for_edit] → Employee resubmits → [pending_manager] Employee cancels → [cancelled] → Employee resubmits → [pending_manager]

3. List Workflow Definitions

Who: Admin or Manager
What: View all workflow definitions in the system. Useful to verify a definition was created or to find its ID.
Phase 2 — Employee Submits a Request

4. Submit a Form (Trigger Workflow)

Who: Employee
What: An employee fills in a leave request form and submits it. The system validates the data against the form schema, creates the submission, and automatically starts the approval workflow. The submission enters pending_manager state.
Real-world example: "I need 3 days of sick leave starting Monday." The employee fills in the reason and number of days, then clicks Submit. Their manager is now notified to review it.
Phase 3 — Manager / HR Reviews

5. Execute Workflow Action

Who: Depends on the action and current state
Actions explained:
• approve — (Manager/Admin) Approve and move to next step. At pending_manager it goes to HR; at pending_hr it becomes fully approved.
• reject — (Manager/Admin) Reject the request permanently. Comment required. Employee can resubmit later.
• return_for_edit — (Manager/Admin) Send back to employee for corrections. E.g., "Please attach a medical certificate." Employee can then resubmit.
• cancel — (Employee/Admin) Employee withdraws their own request. E.g., "I no longer need the day off."
• resubmit — (Employee/Admin) After rejection, cancellation, or return-for-edit, submit a new revision with updated data. Creates a new submission linked to the original.

6. Check Available Actions

Who: Any user viewing a submission
What: Shows which buttons/actions the current user can see based on their role and the submission's current state. In a real UI, this determines which action buttons appear on the submission detail page.
Example: A Manager viewing a pending_manager submission sees: approve, reject, return_for_edit. An Employee sees: cancel.
Phase 4 — Track & Audit

7. Workflow History

Who: Anyone with access to the submission
What: View the full audit trail of a submission — every state change, who did it, and when. This is the timeline an employee sees when checking "What happened to my request?"
Include Revisions: Check this to see the complete history across all resubmissions (revision 1, revision 2, etc.) in a single timeline.

8. My Submissions

Who: Employee
What: The employee's personal dashboard — lists all their own submissions with current status. This is what an employee sees when they open "My Requests" in the app.
Shows: Form name, status (UNDER_REVIEW, APPROVED, REJECTED, CANCELLED, RETURNED), submission date.

9. All Submissions (Admin)

Who: Admin or Manager
What: The admin dashboard — lists ALL submissions from ALL employees. Used by HR or department managers to oversee pending requests, check workload, and monitor approval progress.
Shows: Everything in "My Submissions" plus employee name and email.

10. Submission Revisions

Who: Anyone with access to the submission
What: When an employee resubmits a request (after rejection, cancellation, or return-for-edit), a new revision is created. This shows the full revision chain: original submission (v1), first resubmission (v2), second resubmission (v3), etc.
Real-world example: "My leave request was rejected (v1), I updated the dates and resubmitted (v2), it was approved (v2)." You can enter ANY submission ID in the chain to see all linked revisions.
Test Scenarios

Quick Reference: Test Scenarios

Happy Path (Approved)

1. Employee: Login → Submit form
2. Manager: approve
3. HR/Admin: approve
4. Result: APPROVED
Rejection → Resubmit

1. Employee: Submit form
2. Manager: reject + comment
3. Employee: resubmit with updated data
4. New revision created, back to pending_manager
Return for Edit → Resubmit

1. Employee: Submit form
2. Manager: return_for_edit
3. Employee: resubmit with corrections
4. New revision, back to pending_manager
Cancel → Resubmit

1. Employee: Submit form
2. Employee: cancel
3. Employee: resubmit (optional new data)
4. New revision, back to pending_manager
Cancel (Final)

1. Employee: Submit form
2. Employee: cancel
3. Result: CANCELLED
4. No further actions (unless resubmit)
Full Revision Chain

1. Submit → reject → resubmit (v2)
2. v2: return_for_edit → resubmit (v3)
3. v3: approve → approve
4. Check section 10 for full chain
